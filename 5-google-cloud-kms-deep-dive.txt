================================================================================
1. INTRODUCTION
================================================================================

1.1 What is Google Cloud KMS?
------------------------------

Google Cloud Key Management Service (KMS) is a cloud-hosted service that allows
you to manage cryptographic keys for your cloud services and on-premises
applications. It provides a unified interface to generate, use, rotate, and
destroy various types of keys (symmetric and asymmetric).

Key Value Propositions:
- **Centralized Management:** One place to audit and control all key usage.
- **Hardware Security:** Optional backing by FIPS 140-2 Level 3 HSMs.
- **Identity-First Security:** Integrated with Google Cloud IAM for access.
- **Global Availability:** Distributed across Google's global infrastructure.

1.2 Protection Levels
----------------------

Google Cloud KMS offers three distinct protection levels:

1. **Software (Default):**
   - Keys are stored in secure software enclaves.
   - FIPS 140-2 Level 2 validated.
   - Lowest cost and highest performance for non-regulated tasks.

2. **HSM (Hardware Security Module):**
   - Keys are stored and operations are performed inside physical HSMs.
   - FIPS 140-2 Level 3 validated.
   - **Crucial for Production:** Ensures keys never leave the hardware.

3. **External (EKM):**
   - Keys are stored in an external manager (e.g., Thales, Fortanix) outside
     of Google's infrastructure.
   - Managed via the Cloud External Key Manager (Cloud EKM).

================================================================================
2. CORE CONCEPTS AND ARCHITECTURE
================================================================================

2.1 The Key Hierarchy
----------------------

Google Cloud KMS uses a specific resource hierarchy for organization:

[Project] ──→ [Location] ──→ [KeyRing] ──→ [CryptoKey] ──→ [CryptoKeyVersion]

- **Location:** Geographic region (e.g., us-central1) or multi-region.
- **KeyRing:** A logical grouping of keys. Permissions can be set here.
- **CryptoKey:** The named resource representing a logical key.
- **CryptoKeyVersion:** The actual material. Rotation creates new versions.

2.2 Key Types Supported
------------------------

- **Symmetric Encryption:** AES-256 (GCM mode).
- **Asymmetric Signing:** RSA (2048, 3072, 4096) and ECDSA (P-256, P-384).
- **Asymmetric Encryption:** RSA keys for public-key encryption.
- **MAC Keys:** HMAC for message authentication.

================================================================================
3. PKCS#11 INTEGRATION (THE BRIDGE)
================================================================================

3.1 The Cloud KMS PKCS#11 Library
----------------------------------

To allow standard applications (like OpenSSL, Java, or your TI SoC stack) to use
Cloud KMS, Google provides an open-source PKCS#11 library.

How it works:
1. Application calls a standard PKCS#11 function (e.g., C_Sign).
2. The `libgoogle-kms-pkcs11.so` library intercepts the call.
3. The library translates the call into a Google Cloud KMS REST API request.
4. The request is authenticated via Service Account credentials.
5. The operation is performed in the Cloud HSM, and the result is returned.

3.2 Configuration Requirements
-------------------------------

To use the PKCS#11 bridge, you typically need:
- A configured Google Cloud Service Account with `cloudkms.signerVerifier` roles.
- A YAML configuration file identifying the KeyRing and Location.
- The `pkcs11-tool` or `OpenSSL` configured to load the engine/provider.

================================================================================
4. SECURITY AND COMPLIANCE
================================================================================

4.1 Key Isolation
------------------

In the "HSM" protection level, the private key material is generated inside the
HSM and is never exposed to the host operating system or Google employees in
plaintext. Cryptographic operations are sent to the HSM, and only the results
(signatures/ciphertexts) are returned.

4.2 Auditing and Visibility
----------------------------

Every single interaction with a key is logged in **Cloud Audit Logs**:
- Who used the key?
- When was it used?
- What operation was performed?
- What was the result?

These logs are immutable and can be exported to BigQuery or SIEM tools for
real-time threat detection.

================================================================================
5. PRODUCTION TRADE-OFFS
================================================================================

Advantages:
+ **No Local Secrets:** No private keys are stored on the TI SoC board.
+ **FIPS Compliance:** Meets strict regulatory requirements (FIPS 140-2 L3).
+ **Remote Revocation:** If a board is stolen, you can disable the key in the
  cloud console instantly, rendering the board useless for crypto tasks.

Disadvantages:
- **Internet Dependency:** The board must have a stable connection to sign.
- **Latency:** Adds ~50-200ms of network latency per operation.
- **Cost:** ~$1.00/month per HSM key + API transaction costs.

6. INDUSTRY ADOPTERS
================================================================================

6.1 Who Uses Google Cloud KMS?
-------------------------------

Google Cloud KMS is the standard choice for organizations requiring high-security
key management without the overhead of physical HSM maintenance:

- **Financial Services (FinTech):** Banking apps and payment processors use HSM
  protection for PCI-DSS compliance and transaction signing.
- **Healthcare & Life Sciences:** Securing patient records and health data for
  HIPAA and GDPR compliance.
- **Supply Chain & IoT:** Manufacturers of connected devices (like your TI SoC
  project) for secure firmware updates and device-to-cloud identity.
- **Blockchain & Web3:** Managing hot/cold wallets and signing blockchain
  transactions with FIPS-certified hardware security.
- **Government & Defense:** Meeting strict FedRAMP and FIPS requirements for
  sensitive data handling.

================================================================================
7. THE COMPLETE TECHNICAL STACK (BOTTOM-UP)
================================================================================

7.1 The End-to-End Path
------------------------

For your TI SoC project, the request travels through the following layers:

1. **Hardware (TI SoC Reference Board):**
   - General-purpose CPU and RAM. No specific security hardware required.

2. **Operating System (Embedded Linux):**
   - Provides the kernel network stack, process isolation, and filesystem
     for the PKCS#11 library and configuration.

3. **Application Layer (e.g., OpenSSL, Custom C App):**
   - Requests a cryptographic operation (e.g., "Sign this hash").

4. **The PKCS#11 Bridge (libgoogle-kms-pkcs11.so):**
   - Unlike a TPM driver, there are **no local ioctls** to a hardware device.
   - The library translates the function call into a REST API request in
     user-space.

5. **Secure Internet Connection (TLS 1.3):**
   - The OS network stack creates an encrypted TCP/IP tunnel to Google's
     KMS endpoint (`cloudkms.googleapis.com`) using TLS 1.3.

6. **Google Cloud Frontend (Load Balancer & IAM):**
   - Google's edge infrastructure terminates the TLS and verifies the
     Service Account credentials (IAM) to ensure the request is authorized.

7. **Cloud KMS API Server:**
   - Validates the key resource path and routes the cryptographic request to
     the appropriate HSM pool.

8. **Physical HSM Backend (FIPS 140-2 Level 3):**
   - The final destination. The physical HSM performs the signature or
     decryption inside its secure boundary and returns ONLY the result.
